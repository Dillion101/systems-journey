## What a Function Is in C

A function encapsulates a computation that can be called without knowing how it works internally — only what it takes and what it returns. C has no exponentiation operator, so K&R illustrates functions by writing `power(base, n)` from scratch.

---

## Structure of a Function

```c
return_type function_name(parameter_type param, ...)
{
    // body
    return value;
}
```

- **Return type** — the type of value the function hands back. `int`, `double`, `void` (nothing), etc.
- **Parameters** — local copies of the values passed in. Modifying them inside the function does not affect the caller's variables.
- **return** — exits the function and sends a value back to the caller.

---

## Function Prototypes (Forward Declarations)

The compiler reads C top to bottom in a single pass. If `main` calls a function that is defined *later* in the file, the compiler has not seen its signature yet — it does not know how many arguments it takes, what their types are, or what it returns.

A **prototype** solves this by declaring the function's signature before `main`:

```c
int power(int base, int n);   // prototype — a promise to the compiler

int main() {
    power(2, 5);              // compiler can type-check this call
}

int power(int base, int n)    // definition — fulfils the promise
{
    ...
}
```

The prototype is a contract. The definition must honour it — if the types disagree, the compiler will catch it.

**The two rules:**

| Function defined... | Prototype needed? |
|---|---|
| Before `main` | No — compiler already knows the signature |
| After `main` | Yes — without it, the compiler is flying blind at the call site |

### Prototype vs Definition

The prototype only needs the types — parameter names are optional and do not need to match the definition:

```c
int power(int, int);          // valid prototype — no names
int power(int m, int n);      // also valid — names present but don't need to match definition
```

In practice, keeping names consistent between prototype and definition is cleaner and less confusing.

### Header Files

In larger programs spanning multiple `.c` files, prototypes go in **header files** (`.h`). This is how one `.c` file can call functions defined in another — the header acts as the shared contract between files. Covered in Phase 1 when programs grow beyond a single file.

## Type Consistency Matters

If a function parameter or return type is `int` but the computation produces a `double`, the result is silently truncated. This is a real source of bugs:

```c
int celc = (5.0/9.0) * (fahr - 32);  // right side is double, stored as int — truncated
```

`fahr_celc(100)` would return `37`, not `37.77`. The fix is to match types throughout:

```c
double fahr_celc(double fahr) {
    double celc = (5.0/9.0) * (fahr - 32);
    return celc;
}
```

**Rule:** if the computation involves fractional values, the parameters, local variables, and return type should all be `double`.

## Exercises

### Exercise 1-15 — Rewrite temperature conversion using a function

**Task:** Rewrite the Fahrenheit-to-Celsius table from Section 1.2 to use a function for conversion.

**First attempt (bug):**

```c
int fahr_celc(int fahr) {
    int celc = (5.0/9.0) * (fahr - 32);
    return celc;
}
```

Problem: `celc` is `int` but the right-hand side produces a `double`. The fractional part is silently truncated. `fahr_celc(100)` returns `37` instead of `37.77`.

**Corrected solution:**

```c
#include <stdio.h>

double fahr_celc(double fahr);

double fahr_celc(double fahr)
{
    double celc = (5.0/9.0) * (fahr - 32);
    return celc;
}

int main()
{
    double fahrenheit = 100;
    printf("%.3f F is equivalent to %.3f Celsius\n", fahrenheit, fahr_celc(fahrenheit));
    return 0;
}
```

**What changed:**
- `int` → `double` throughout — parameter, local variable, return type, and `main`'s variable all consistent
- `%d` → `%.3f` — correct format specifier for `double`, three decimal places

**On prototypes:** the prototype `double fahr_celc(double fahr)` at the top is redundant here because the definition appears *before* `main`. Removing it still compiles. It was retained for practice. In `functions.c`, the prototype for `power` is doing real work because the definition comes after `main`.
