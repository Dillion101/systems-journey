# K&R Section 1.10 — External Variables and Scope
*Date: May 7, 2026*

---

## What External Variables Are

Every variable declared so far has been *local* — it exists only inside the function that declares it, and dies when the function returns. External variables are different: they are declared *outside* any function, live for the entire execution of the program, and can be accessed by any function that knows about them.

```c
int max;
char line[MAXLINE];
char longest[MAXLINE];
```

These are external variables. They sit at file scope, not inside `main` or any other function.

---

## Scope

*Scope* is the region of code where a name is visible.

- A **local** variable's scope is the block `{}` it was declared in. Nothing outside that block can see it.
- An **external** variable's scope starts at its definition and extends to the end of the file.

If a function wants to use an external variable defined *later* in the same file, or defined in a *different* file, it must declare it with `extern`:

```c
extern int max;
extern char longest[];
```

This tells the compiler "this variable exists somewhere — don't allocate it here, just let me use it." The `extern` is optional when the definition appears earlier in the same file, but it is never wrong to include it for clarity.

---

## Definition vs Declaration

This distinction matters and K&R are precise about it:

- A **definition** allocates storage. There must be exactly one.
- A **declaration** announces the type. There can be many.

```c
// definition — storage is allocated here
int max;

// declaration — no storage, just says "max exists"
extern int max;
```

Defining the same variable twice is a linker error. Declaring it many times is fine.

## The Longest-Line Program — Specialised (External) Version

This is the version in `longest_line.c`. It moves `line`, `longest`, and `max` to external scope so that `my_getline` and `copy` can access them directly, without needing parameters.

### Tradeoffs

Using external variables removes the need to pass data through function arguments. But it comes at a cost:

- Functions are no longer self-contained — they silently depend on globals
- The same function cannot be reused in a different program without bringing its globals along
- Reasoning about the program gets harder as it grows, because any function could be modifying shared state

K&R recommend preferring local variables and explicit arguments. External variables are appropriate for data that genuinely needs to be shared across many functions with no clear "owner."

### `my_getline(void)` — specialised version

Reads into the *external* `line[]` directly. No parameters — it knows where to put the data because `line` is global.

```c
int my_getline(void)
{
    int c, i;
    extern char line[];
    for (i = 0; i < MAXLINE - 1 && ((c = getchar()) != EOF) && c != '\n'; ++i)
        line[i] = c;
    if (c == '\n') {
        line[i] = c;
        ++i;
    }
    line[i] = '\0';
    return i;
}
```

Compare to the parameterised version from 1.9 — the logic is identical. The only difference is where `line` comes from.

### `copy(void)` — specialised version

Copies from external `line[]` into external `longest[]`. Again, no parameters.

```c
void copy(void)
{
    int i;
    extern char line[], longest[];
    i = 0;
    while ((longest[i] = line[i]) != '\0')
        ++i;
}
```

### `main()`

```c
int main()
{
    int len;
    extern int max;
    extern char longest[];
    max = 0;
    while ((len = my_getline()) > 0)
        if (len > max) {
            max = len;
            copy();
        }
    if (max > 0)
        printf("%s", longest);
    return 0;
}
```

`len` stays local — it is only needed inside `main`. `max` and `longest` are external because `copy` needs to update them.

---

## Key Pattern: `extern` Inside a Function

The `extern` declarations inside `main`, `my_getline`, and `copy` are redundant here — the variables are already defined at file scope above all of them. But they act as documentation: each function explicitly announces which globals it depends on. This is a good habit in larger programs where the globals may not be immediately visible at the top of the file.

---

## Why Automatic Variables Don't Persist

An *automatic* variable (the default for locals) is created on the stack when the function is called and destroyed when it returns. This means two calls to the same function each get their own fresh copy — no communication happens between calls through locals.

External variables persist because they are allocated in a separate region of memory (the data segment) that exists for the lifetime of the process. This is why they can carry state between function calls.

### Exercise 1-20 — `detab`: replace tabs with spaces

**Problem:** Replace each tab with the correct number of spaces to reach the next tab stop. Tab stops are fixed at every `n` columns.

**Decision — should `n` be a variable or a symbolic parameter?** A symbolic parameter (`#define`) is the right call here. Tab size is a compile-time constant, not something that changes per run. `#define TABSIZE 8` makes it easy to change once without touching the logic.

```c
#include <stdio.h>
#define TABSIZE 8

int main() {
    int c, col;
    col = 0;

    while ((c = getchar()) != EOF) {
        if (c == '\t') {
            int spaces = TABSIZE - (col % TABSIZE);
            for (int i = 0; i < spaces; i++) {
                printf(" ");
            }
            col += spaces;
        } else if (c == '\n') {
            printf("\n");
            col = 0;
        } else {
            printf("%c", c);
            col++;
        }
    }
    return 0;
}
```

**Key insight:** `TABSIZE - (col % TABSIZE)` always gives the number of spaces to the next stop. When `col` is already on a tab stop (`col % TABSIZE == 0`), this correctly gives a full `TABSIZE` spaces — the tab moves to the *next* stop, not the current one.

### Exercise 1-21 — `entab`: replace spaces with minimum tabs and blanks

**Problem:** The inverse of `detab`. Replace runs of spaces with the fewest tabs and blanks that produce the same visual spacing. Same tab stops.

**Preference decision — tab or single blank when either suffices?** Single blank. A tab when only one space is needed adds no compression value and makes the output harder to read.

The key idea is a *bucket* — accumulate spaces, and only emit a tab once the bucket reaches the next tab stop exactly. If a non-space character arrives while the bucket has pending spaces, flush them as individual spaces first.

```c
#include <stdio.h>
#define TABSIZE 8

int main()
{
    int c;
    int bucket = 0;
    int col = 0;

    while ((c = getchar()) != EOF)
    {
        int nextTab = TABSIZE - (col % TABSIZE);

        if (c == ' ')
        {
            bucket++;
            if (bucket == nextTab)
            {
                printf("\t");
                col = col + nextTab;
                bucket = 0;
            }
            else
            {
                col++;
            }
        }
        else if (c == '\n')
        {
            for (int i = 0; i < bucket; i++)
                printf(" ");
            printf("\n");
            col = bucket = 0;
        }
        else
        {
            for (int i = 0; i < bucket; i++)
                printf(" ");
            printf("%c", c);
            bucket = 0;
            col++;
        }
    }
}
```

**Key insight:** `col` tracks where we *actually are* in the output, including spaces already printed. `bucket` holds spaces that have been counted but not yet emitted — they are pending until we know whether they become a tab or stay as spaces.
