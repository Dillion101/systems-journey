# K&R Section 1.5 — Character Input and Output
*Date: April 16–18, 2026*

---

## The Model: Text Streams

C treats all input and output as streams of characters. Whether you are reading from the keyboard, a file, or a pipe, the program sees the same thing: a sequence of characters arriving one at a time. This is the UNIX philosophy made concrete in the language itself.

---

## getchar() and putchar()

The two most fundamental I/O functions in C:

- `getchar()` — reads one character from standard input and returns it as an `int`
- `putchar(c)` — writes one character to standard output

```c
int c;
c = getchar();
putchar(c);
```

`c` is declared as `int`, not `char` — see EOF section below for why.

---

## EOF — End of File

EOF is a symbolic constant defined in `<stdio.h>`. Its value is `-1`.

It is the sentinel value that `getchar()` returns when there is no more input — either because you have reached the end of a file, or because the user typed `Ctrl+D` on Linux/macOS (`Ctrl+Z` on Windows).

This is why `c` must be `int` and not `char`. A `char` typically holds values 0–255. EOF is `-1`, which falls outside that range. If you store `getchar()`'s return value in a `char`, you may never correctly detect EOF — the program could loop forever.

**Rule: always store the return value of `getchar()` in an `int`.**

---

## The Standard Read Loop

```c
int c;
while ((c = getchar()) != EOF)
    putchar(c);
```

Two things to understand:

1. The assignment `c = getchar()` happens inside the condition. The parentheses around it are required — without them, `!=` binds tighter than `=` and the logic breaks.
2. The loop body runs zero or more times. If the very first character is EOF, the body never executes.

## Section 1.5.2 — Character Counting

Introduces `long` — used instead of `int` when a count could overflow a 32-bit integer.

```c
long nc;
nc = 0;
while (getchar() != EOF)
    ++nc;
printf("%ld\n", nc);
```

- `%ld` is required for `long` — using `%d` is undefined behaviour
- `getchar()` return value is not stored since we only care about counting

A second version using `for` with an empty body:

```c
for (nc = 0; getchar() != EOF; ++nc)
    ;
```

The isolated `;` is a null statement — valid C that satisfies the grammatical requirement for a loop body when all work is in the loop header.

---

## Section 1.5.3 — Line Counting

Lines are counted by counting `\n` characters. Every UNIX text line ends with `\n`.

```c
int c, nl;
nl = 0;
while ((c = getchar()) != EOF)
    if (c == '\n')
        ++nl;
printf("%d\n", nl);
```

- `'\n'` is a character constant — a single character, value 10 in ASCII
- Single quotes = character constants; double quotes = string literals. Not interchangeable.

Functionally equivalent to `wc -l` on UNIX.

## Section 1.5.4 — Word Counting

Combines character, line, and word counting. The key challenge: defining a "word" in a character stream requires a **state machine**.

```c
#define IN  1  /* inside a word */
#define OUT 0  /* outside a word */

int c, nl, nw, nc, state;
state = OUT;
nl = nw = nc = 0;

while ((c = getchar()) != EOF) {
    ++nc;
    if (c == '\n')
        ++nl;
    if (c == ' ' || c == '\n' || c == '\t')
        state = OUT;
    else if (state == OUT) {
        state = IN;
        ++nw;
    }
}
```

Key points:

- `nw` increments only on the `OUT → IN` transition — not on every non-whitespace character. A five-letter word crosses that boundary once, not five times.
- The two `if` blocks are intentionally separate (not `else if`). A newline must both increment `nl` AND set `state = OUT`.
- `IN` and `OUT` as named constants make the state transitions readable — `state == OUT` is immediately meaningful; `state == 0` requires mental lookup.

### What this actually is: a Finite State Machine

```
         whitespace
    ┌──────────────────┐
    ▼                  │
  OUT ──────────────► IN
       non-whitespace
         (count word)
```

Two states, transitions on input. Lexers, parsers, network protocol handlers, and regex engines all use this same pattern at their core.

### The `||` trap — why `c == ' ' || '\n' || '\t'` is always true

```c
if (c == ' ' || '\n' || '\t')   /* WRONG */
```

C parses this as `(c == ' ') || ('\n') || ('\t')`. `'\n'` is integer 10, `'\t'` is integer 9 — both truthy. The expression is always true regardless of `c`. Every character gets treated as whitespace.

```c
if (c == ' ' || c == '\n' || c == '\t')   /* correct */
```

C has no `in` operator. Every comparison is binary. Write `c ==` every time. The compiler will not warn you about the broken version.

## Exercises

### Exercise 1-6 — Verify `getchar() != EOF` is 0 or 1

```c
#include <stdio.h>
int main(void) {
    int c;
    while ((c = getchar()) != EOF)
        printf("%d\n", c != EOF);
    return 0;
}
```

Comparisons in C always produce exactly `1` (true) or `0` (false). `c != EOF` is `1` for all valid characters inside the loop.

### Exercise 1-7 — Print the value of EOF

```c
#include <stdio.h>
int main() {
    printf("EOF value: %d\n", EOF);
}
```

Output: `EOF value: -1`. EOF is the integer `-1`, not a character. No valid character has this value, which is why it works as a sentinel.

### Exercise 1-8 — Count blanks, tabs, and newlines

```c
#include <stdio.h>
int main() {
    int c, nl, tc, bc;
    nl = tc = bc = 0;
    while ((c = getchar()) != EOF)
        if (c == '\n')
            nl++;
        else if (c == '\t')
            tc++;
        else if (c == ' ')
            bc++;
    printf("New Line Count: %d, Tab Count: %d, Blank Count: %d\n", nl, tc, bc);
}
```

Chained `if/else if` is correct here — each character is exactly one thing, so branches are mutually exclusive.

### Exercise 1-9 — Collapse multiple blanks into one

```c
#include <stdio.h>
int main() {
    int c, prev_was_space;
    prev_was_space = 0;
    while ((c = getchar()) != EOF) {
        if (c == ' ' && prev_was_space == ' ') {}
        else
            printf("%c", c);
        prev_was_space = c;
    }
}
```

Demonstrates state tracking across loop iterations. `prev_was_space` holds the previous character so the current iteration can make a context-aware decision. This "remember what you just saw" pattern appears constantly in stream processing.

### Exercise 1-10 — Make tabs and backspaces visible

```c
#include <stdio.h>
int main() {
    int c;
    while ((c = getchar()) != EOF) {
        if (c == '\t')
            printf("\\t");
        else if (c == '\b')
            printf("\\b");
        else if (c == '\\')
            printf("\\\\");
        else
            putchar(c);
    }
    return 0;
}
```

When `c == '\t'`, the character is a real tab (ASCII 9). To print the two visible characters `\t`, output `"\\t"` — the double backslash produces a literal `\`. This transformation pattern is foundational in parsers and formatters.

### Exercise 1-11 — How would you test the word count program?

Test cases that stress the state machine boundaries:

- Empty input — all counts should be zero
- Only whitespace — word count should be zero
- Single word with no trailing newline — must still be counted
- Multiple consecutive spaces — `"hello   world"` = 2 words
- Leading and trailing whitespace — `"  hello  "` = 1 word
- Tabs between words — `"hello\tworld"` = 2 words
- Mixed whitespace — combinations of spaces, tabs, newlines

Bugs in state machines appear at transitions, not in the middle of normal input.

### Exercise 1-12 — Print input one word per line

```c
#include <stdio.h>

#define IN  1
#define OUT 0

int main() {
    int c, state;
    state = OUT;
    while ((c = getchar()) != EOF) {
        if (c == ' ' || c == '\n' || c == '\t') {
            if (state == IN)
                printf("\n");
            state = OUT;
        } else {
            putchar(c);
            state = IN;
        }
    }
}
```

Print a newline only when whitespace is encountered *after* being inside a word (`state == IN`). This prevents blank lines for consecutive whitespace.

**Process note:** Solved only after writing pseudocode on paper first and manually tracing through multi-word, multi-space input before writing any code. The lesson: pseudocode and manual tracing are not a beginner crutch — they are how good engineers avoid writing code that needs to be thrown away.
